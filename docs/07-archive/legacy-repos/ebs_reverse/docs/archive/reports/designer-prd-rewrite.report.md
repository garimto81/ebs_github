## 4. Lessons Learned

### What Went Well

1. **Pre-planning Excellence**: Detailed plan document (Plan phase) prevented ambiguity and false starts. All 3 implementation tasks executed with zero rework except for count verification.

2. **Modular Content Transformation**: Breaking 149 features into 7 user scenarios (vs. 8 technical categories) proved highly effective for readability. Each scenario immediately answers "When would a user need this?"

3. **Visual-First Approach**: Mermaid diagrams (9 total) provided clarity that text alone could not. Especially effective:
   - Journey diagram (viewer experience)
   - Sequence diagram (card tag → display flow)
   - Quadrant chart (Impact vs. Effort priority matrix)

4. **Existing Asset Reuse**: Leveraged 14 existing images (mockups + annotated screenshots) instead of creating new diagrams. Time-efficient and improved visual consistency.

5. **Perspective Shift Simplicity**: Removing technical depth (RFID UID parsing, GPIO pinmaps) was straightforward once guardrails were clear. No ambiguity about what "no code blocks" meant.

### Areas for Improvement

1. **Feature Checklist Maintenance**: The summary table (line 16) drifted from actual feature list. v4.1.0 added 30 features but didn't update summary counts.
   - **Recommendation**: Automated sync mechanism (e.g., Python script that counts categories and updates table)
   - **Impact**: Would have prevented 2nd iteration

2. **P0/P1/P2 Definition Precision**: Discrepancy arose because Checklist.md uses flat P0/P1/P2 assignment, but Designer PRD groups features by scenario. The same feature might contribute to multiple scenarios.
   - **Recommendation**: Create explicit mapping document: Feature ID → P0/P1/P2 + Scenario Group
   - **Effort**: Low (30 min in Excel)

3. **Version Numbering for Checkpoints**: Designer PRD v1.0.0 was final, but Feature Checklist v4.1.0 was outdated. Need clear versioning rule.
   - **Recommendation**: Always sync doc versions when publishing (both should be v1.4.1 or similar)

### To Apply Next Time

1. **Validate Summary Tables Early**: Before starting writing, verify that summary tables (totals, counts, percentages) match detailed source data. Could have caught the P0/P1/P2 discrepancy before iteration 1.

2. **Journey Maps for Stakeholders**: Designer-targeted docs benefit from Mermaid journey diagrams (not just text scenarios). Include at least 2 (operator + user).

3. **Mockup-Centric Structure**: Rather than describing screens textually, use existing wireframes/screenshots as the primary reference and text as annotation. Improves buy-in from design teams.

4. **Scenario-Based Grouping**: When transforming technical docs for non-technical audiences, reorganize by "user moment" (When do they need X?) instead of technical categories. Drastically improves clarity.

5. **Two-Pass Review**: First pass = content correctness (all features present?). Second pass = consistency (do summaries match details?). Prevented missing the count drift.
