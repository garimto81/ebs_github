## 7. 교훈 및 개선사항

### 7.1 무엇이 잘 되었는가 (Keep)

1. **내러티브 기반 문서가 정보 나열형보다 훨씬 효과적**
   - 읽는 사람이 자연스럽게 필요성 이해
   - 포커 방송 특수성부터 시작하는 스토리텔링

2. **Ralplan 3자 합의가 문서 구조 결정에 매우 효과적**
   - Planner: 요구사항 분석
   - Architect: 기술 관점
   - Critic: 비판적 검토
   - → 균형 잡힌 구조 도출

3. **역공학 데이터의 명확한 분리**
   - PS-001~013 제거로 PRD 깔끔화
   - Feature Checklist로 위임으로 각 문서의 역할 명확화

4. **Master PRD를 단일 진실 공급원으로 명확히**
   - Phase 번호 일관성 확보
   - 하위 문서는 항상 Master PRD 참조

5. **500줄 제약이 좋은 discipline**
   - 불필요한 정보 누적 방지
   - 본질적인 내용만 남기도록 강제

### 7.2 개선할 점 (Problem)

1. **기술 정확성 재검증**
   - RFID 안테나: 26개 vs 22개 vs 21개?
   - 승인 전 한 번 더 확인 필요

2. **두 레포(ebs vs ebs_reverse) 간 참조 경로**
   - Google Drive 동기화 시 경로 문제 가능성
   - 명시적 경로 규칙 문서화 필요

3. **Feature Checklist와 PRD 동기화**
   - 장기간 따로 진화하면 불일치 가능
   - 동기화 프로토콜 수립 필요

### 7.3 다음번에 적용할 사항 (Try)

1. **다층 문서 전략 공식화**
   - Level 1 (PRD): 비즈니스 맥락 + 아키텍처 개요
   - Level 2 (Feature Checklist): 149개 기능 상세
   - Level 3 (기술 설계): 아키텍처, API 명세
   - Level 4 (역공학): 상세 프로토콜, 네트워크 명령

2. **Ralplan을 복잡도 5/5 정석 프로세스로**
   - 이번 성공 사례를 기준으로 정석화
   - 다음 5/5 작업 시 반복 적용

3. **Gap Detection 체크리스트 재사용**
   - 7개 PASS 항목이 모두 명확하게 검증됨
   - 유사 문서 작업 시 동일 체크리스트 사용

4. **Master PRD 정합성 자동 검증**
   - YAML frontmatter로 참조 관계 정의
   - 자동 검증 도구 개발 고려
