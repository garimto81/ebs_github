#### 1. 기획자/개발자 경계는 명확한 프레임으로 정의 가능

원래 경계가 모호해 보였지만, **WHAT/WHY (기획자) vs HOW (개발자)** 프레임을 적용하면 자동으로 구분됨.

**적용 예시**:
- "WebSocket 실시간 연결" → WHAT: 실시간 데이터, HOW: WebSocket (기술)
- "카드 인식 200ms" → WHAT: 인식 속도 (기획자), HOW: 200ms 달성 방법 (개발자)

#### 2. "방송 업계 준기술 용어"도 문맥에 따라 제거 필요

처음에는 "파이프라인", "버퍼" 같은 업계 용어를 허용했지만, **기획자 문서에서는 더 단순한 표현이 필수**.

**이유**: 기획자가 방송 업계 용어를 모를 수 있으므로, 프로젝트 공통 용어(RFID, Equity, Chroma Key)는 허용해도 순수 기술 표현은 제거 필요.

#### 3. Mermaid 다이어그램의 participant 수를 줄이면 기획자 가독성이 크게 향상

- **8개 participant** (기술 수준): 시스템 엔지니어는 이해하지만 기획자는 혼란
- **4개 participant** (기획자 수준): 역할이 명확하고 정보 흐름이 직관적

**핵심**: 다이어그램은 "얼마나 정확한가"보다 "얼마나 이해하기 쉬운가"가 기획자 문서에서는 더 중요.
