# PokerGFX — 포커 방송 그래픽 시스템 기획서

**Version**: 4.0.0
**Date**: 2026-02-15

> 이 문서는 PokerGFX라는 포커 방송 그래픽 시스템이 무엇인지, 왜 우리가 동일한 시스템을 직접 만들어야 하는지, 그리고 이 시스템이 어떻게 동작하는지를 설명합니다.

## 1. 포커 방송의 세계

### 일반 스포츠와 포커의 결정적 차이

스포츠 방송은 **카메라가 곧 정보**다. 축구에서 카메라는 공의 위치, 선수의 움직임, 골 장면을 모두 촬영하고, 그래픽 시스템은 그 위에 점수판과 선수 이름을 얹어주기만 하면 된다. 야구, 농구, 테니스도 마찬가지다. 카메라만 있으면 방송에 필요한 모든 원시 정보를 얻을 수 있다.

포커는 근본적으로 다르다. 포커에서 가장 중요한 정보는 각 플레이어의 **홀카드**(비공개 카드)인데, 카드는 뒤집어져 있어서 카메라로 촬영해도 무슨 카드인지 알 수 없다. 시청자가 각 플레이어의 카드를 알아야 "누가 유리한가"를 판단하고 경기에 몰입할 수 있는데, 카메라만으로는 이 정보를 얻을 수 없다.

![일반 스포츠 vs 포커 — 정보 획득 방식의 차이](images/prd/prd-info-comparison.png)

정리하면 이렇다:
- **일반 스포츠**: 경기장 → 카메라 → 영상 + 정보 → 그래픽 오버레이 → 방송
- **포커**: 테이블 → RFID 리더 → 카드 정보 → 서버 처리 → 그래픽 오버레이 → 방송. 카메라는 영상만 제공한다.

포커 방송에서는 카메라가 아닌 **다른 방법**으로 카드 정보를 얻어야 한다. 이 문제를 해결하기 위해 카드 인식 기술이 발전해왔다.

### 카드 인식 기술의 변천사

![카드 인식 기술 변천사 — 유리판 카메라에서 RFID로](images/prd/prd-card-tech-evolution.png)

**1세대: 유리판 카메라 방식**

테이블의 플레이어 앞에 유리판을 설치하고, 그 아래에 소형 카메라를 배치했다. 플레이어가 카드를 유리판 위에서 살짝 들어올려 확인하면 아래 카메라가 카드 앞면을 촬영하는 방식이다.

한계가 명확했다. 플레이어가 의도적으로 카드를 카메라 위에 보여줘야 하고, 조명이나 각도에 따라 인식이 실패하며, 설치가 복잡했다.

**2세대: RFID 방식 (현재)**

카드 안에 RFID 태그를 삽입하고, 테이블에 RFID 리더를 매립한다. 플레이어 앞에 카드가 놓이기만 하면 리더가 자동으로 카드를 인식한다. 뒤집어 놓아도, 겹쳐 놓아도 인식이 가능하다.

이 방식이 안정화되면서 현재 모든 주요 포커 방송이 RFID 기반으로 운영되고 있다.

**카드를 읽고, 게임을 추적하고, 승률을 계산하고, 화면에 그래픽으로 표시하는 이 전체 과정을 하나의 시스템으로 관리하는 것이 PokerGFX다.**

## 2. 전체 시스템 구조

### 3계층 아키텍처

PokerGFX는 크게 세 가지 계층으로 구성된다.

![시스템 3계층 구조 — 인식, 처리, 화면](images/prd/prd-3layer-architecture.png)

| 계층 | 역할 | 구성 요소 |
|------|------|----------|
| **인식** | 카드의 물리적 인식 | RFID 리더 12대 (좌석 10 + 보드 1 + Muck 1) |
| **처리** | 게임 로직, 승률, 통계 계산 | GFX Server, 핸드 평가기 17종, 통계 엔진 |
| **화면** | 방송 화면 출력 | Dual Canvas (Live + Delayed), NDI/HDMI 출력 |

### 7개 앱 생태계

시스템 전체는 서버 1개와 클라이언트 6개, 총 7개 앱으로 구성된다.

![7개 앱 생태계 — 누가 무엇을 사용하는가](images/prd/prd-7app-ecosystem.png)

| 앱 | 역할 | 사용자 |
|----|------|--------|
| **GFX Server** | 방송 미리보기, 7개 설정 탭, 출력 관리 | 방송 엔지니어 |
| **Action Tracker** | 실시간 핸드 추적, 베팅 액션 입력 | 테이블 운영자 |
| **Hand Evaluation** | 승률과 아웃츠 실시간 계산 | 자동 (내부 서비스) |
| **Action Clock** | 플레이어 시간 제한 카운트다운 | 방송 화면 표시 |
| **StreamDeck** | 물리 버튼으로 방송 제어 | 방송 엔지니어 |
| **Pipcap** | 비디오 소스 캡처 | 자동 (내부 서비스) |
| **Commentary Booth** | 해설자 원격 접속, 홀카드 전용 뷰어 | 해설자 |

GFX Server는 메인 윈도우 + 7개 탭 (System, Sources, Outputs, GFX1, GFX2, GFX3, Commentary) + 3개 독립 편집기 (Skin Editor, Board Editor, Player Editor)로 구성된 11개 화면을 갖고 있다.

### RFID 테이블 배치

![RFID 리더 12대 테이블 배치](images/prd/prd-rfid-table-layout.png)

하나의 포커 테이블에는 RFID 리더 12대가 매립된다:
- **좌석 리더 10대**: 각 플레이어 앞에 1대씩, 홀카드를 자동 인식
- **보드 리더 1대**: 테이블 중앙, 커뮤니티 카드 5장을 인식
- **Muck 리더 1대**: 딜러 앞, 폴드/버린 카드를 인식

리더는 테이블 펠트 아래에 완전히 매립되어 플레이어에게 보이지 않는다.

## 3. 운영자가 하는 일

### 핵심 원칙

**운영자는 카드를 입력하지 않는다.** RFID가 자동으로 인식한다. 운영자는 베팅 액션만 입력한다.

### Action Tracker 화면

![Action Tracker — 운영자가 사용하는 화면](images/prd/prd-action-tracker.png)

Action Tracker는 운영자의 주 작업 도구다. 화면 구성은 다음과 같다:

- **이벤트 정보 바**: 대회명, 게임 유형, 블라인드 레벨
- **플레이어 테이블**: 좌석별 이름, 칩, 홀카드(RFID 자동), 상태, 최근 액션
- **보드 영역**: 커뮤니티 카드 5장 (RFID 자동) + 메인팟 금액
- **액션 버튼 7종**: FOLD, CHECK, CALL, BET, RAISE, ALL-IN, POST
- **특수 버튼**: UNDO (최대 5단계), MISS DEAL, RUN IT 2x, CHOP, TAG HAND

운영자의 역할은 7개 액션 버튼 중 하나를 누르는 것이 전부다. 카드 인식, 팟 계산, 승률 계산, 화면 갱신은 모두 시스템이 자동으로 처리한다.

### 핸드 1회 진행 시퀀스

딜러가 카드를 배분하는 순간부터 승자가 결정되기까지의 전체 흐름이다.

![핸드 1회 진행 시퀀스 — 사람이 하는 일 vs 시스템이 하는 일](images/prd/prd-hand-sequence.png)

| Phase | 딜러 | RFID | 운영자 | 서버 | 방송 화면 |
|-------|------|------|--------|------|----------|
| **1. 카드 배분** | 카드 배분 | 자동 인식 | — | 카드 저장 | 홀카드 표시 |
| **2. 베팅 라운드** | — | — | BET/CALL 입력 | 팟·승률 계산 | 액션·승률 갱신 |
| **3. 보드 카드** | 보드 오픈 | 자동 인식 | — | 승률 재계산 | 보드·승률 갱신 |
| **4. 승자 결정** | — | — | — | 핸드 비교·판정 | 위닝 핸드 강조 |

**사람이 하는 일은 딱 두 가지**: 딜러의 "카드 배분"과 운영자의 "액션 버튼 누르기". 나머지는 전부 자동이다.

### 방송 준비 워크플로우

방송 시작 전에는 다음 순서로 준비한다:

1. 대회/이벤트 설정 (이름, 게임 타입, 블라인드)
2. 플레이어 좌석 배치 (이름, 스택, 포지션)
3. RFID 덱 등록 (52장 일괄 스캔)
4. 출력 확인 (Live/Delayed 파이프라인 점검)

## 4. 시청자가 보는 화면

### 방송 오버레이

카메라 영상 위에 PokerGFX가 실시간으로 오버레이하는 그래픽 요소가 시청자에게 보이는 화면이다.

![방송 오버레이 — 시청자에게 보이는 8개 그래픽 요소](images/prd/prd-broadcast-overlay.png)

플레이어 오버레이는 8개 요소로 구성된다:

| # | 요소 | 설명 |
|:-:|------|------|
| ① | 국기 | 플레이어 국적 |
| ② | 이름 | 플레이어 이름 |
| ③ | 홀카드 | RFID로 인식된 비공개 카드 (게임에 따라 2~7장) |
| ④ | 스택 | 보유 칩 금액 (K/M 축약 표시) |
| ⑤ | 액션 | 최근 베팅 행동 (FOLD, CALL $500, RAISE $3,000 등) |
| ⑥ | 승률 바 | Monte Carlo 시뮬레이션 기반 실시간 승률 |
| ⑦ | 포지션 | 테이블 위치 (D, SB, BB, UTG 등) |
| ⑧ | 프로필 사진 | 플레이어 얼굴 사진 |

이 외에도 테이블 중앙에는 커뮤니티 카드 5장, 메인팟과 사이드팟 금액, 이벤트 배너(대회명, 블라인드 레벨)가 표시된다.

### 애니메이션

화면 전환과 상태 변화를 자연스럽게 보여주기 위해 16종의 애니메이션 상태와 11개 애니메이션 클래스가 사용된다. 주요 애니메이션:
- 카드 등장/소멸
- 올인 시 승률 바 전체 표시
- 폴드 시 플레이어 그레이아웃
- 승자 결정 시 위닝 핸드 강조
- 팟 금액 증가 카운트업

그래픽 요소는 4가지 타입으로 분류된다: image (41개 속성), text (52개 속성), pip (12개 속성, 카드 문양), border (테두리).

### Dual Canvas — 왜 화면이 2개 필요한가

![Dual Canvas — 선수 보호와 시청자 경험의 양립](images/prd/prd-dual-canvas.png)

같은 게임 데이터를 2개의 독립된 화면으로 출력한다.

| Canvas | 용도 | 홀카드 | 대상 | 출력 |
|--------|------|:------:|------|------|
| **Live** | 경기장 모니터 | 숨김 | 선수, 관객 | NDI/HDMI |
| **Delayed** | 방송 송출 | 지연 후 표시 | 시청자 | NDI/HDMI |

**왜 2개인가?** 단일 화면이라면 두 가지 문제 중 하나를 피할 수 없다:
- 경기장 모니터에 홀카드가 표시되면 선수나 관객이 상대의 카드를 볼 수 있어 게임이 성립하지 않는다.
- 반대로 홀카드를 숨기면 시청자가 카드 뒷면만 보게 되어 방송이 성립하지 않는다.

Dual Canvas는 Live Canvas로 경기장 공정성을 보장하고, Delayed Canvas로 시청자 경험을 확보한다. Trustless 모드에서는 Live Canvas에서 승률 표시까지 비활성화한다.

## 5. 왜 이 시스템을 만드는가

### 핵심 자산 내재화

현재 방송에 사용하는 PokerGFX는 외부 시스템이다. 핸드 히스토리와 플레이어 통계 — 방송 품질과 콘텐츠 기획의 핵심 자산 — 이 외부에 있어서 우리가 통제할 수 없다. PokerGFX가 경쟁사에 인수된 상황에서, 라이선스 갱신 거부나 가격 인상 같은 비즈니스 리스크가 현실적 위협이 되었다.

PokerGFX는 구매 대상이 아니라 **소프트웨어 벤치마크/복제 대상**이다.

### 운영 효율화

현재 30명이 투입되는 방송 인력을 자막 연출 자동화를 통해 15~20명으로 줄일 수 있다.

### WSOP+ 플랫폼 통합

자체 시스템이 있어야 대회 데이터베이스에서 방송 그래픽까지의 파이프라인을 자유롭게 설계할 수 있다:
- 플레이어 정보 자동 로드 (수동 입력 제거)
- 대회별 스킨 자동 적용
- 다중 테이블 동시 방송
- 커스텀 통계/분석 지표

### 타겟 사용자 5종

| 역할 | 핵심 목표 | 주요 사용 화면 |
|------|----------|---------------|
| 시청자 | 몰입감 있는 시청 (실시간 홀카드, 승률) | 방송 오버레이 |
| 테이블 운영자 | 최소 클릭으로 액션 입력 | Action Tracker |
| 방송 엔지니어 | 출력 품질 제어 (Dual Canvas, 스킨) | GFX Server |
| 해설자 | 실시간 홀카드 확인 (보안 분리) | Commentary Booth |
| 콘텐츠 PD | 하이라이트, 통계 활용 | 핸드 히스토리 |

## 6. 게임 종류와 운영 모드

### 22개 포커 게임 변형

PokerGFX는 3개 계열, 22개 포커 변형을 지원한다.

**Community Card 계열 (13개)** — 공유 보드 카드를 사용하는 게임

| 게임 | 홀카드 | 특징 |
|------|:------:|------|
| Texas Hold'em | 2장 | 가장 대중적인 포커 |
| 6+ Hold'em (2종) | 2장 | 2~5를 제거한 36장 덱 |
| Pineapple | 3장→2장 | Flop 전 1장 버림 |
| Omaha (2종) | 4장 | 반드시 홀카드 2장 사용 |
| Five Card Omaha (2종) | 5장 | 5장 중 2장 사용 |
| Six Card Omaha (2종) | 6장 | 6장 중 2장 사용 |
| Courchevel (2종) | 5장 | 첫 Flop 카드 미리 공개 |

**Draw 계열 (6개)** — 카드를 교환하는 게임

| 게임 | 카드 | 교환 | 특징 |
|------|:----:|:----:|------|
| Five Card Draw | 5장 | 1회 | 기본 Draw |
| 2-7 Single/Triple Draw | 5장 | 1~3회 | Lowball |
| A-5 Triple Draw | 5장 | 3회 | A-5 Lowball |
| Badugi | 4장 | 3회 | 4장 Lowball, 서로 다른 무늬가 유리 |
| Badeucy/Badacey | 5장 | 3회 | 혼합 게임 |

**Stud 계열 (3개)** — 일부 카드가 앞면으로 공개되는 게임

| 게임 | 카드 | 특징 |
|------|:----:|------|
| 7-Card Stud | 7장 | 3장 비공개 + 4장 공개 |
| Stud Hi-Lo 8 | 7장 | 팟 분할 |
| Razz | 7장 | Lowball Stud |

### 운영 모드

| 모드 | 설명 |
|------|------|
| Realtime | 홀카드와 승률을 실시간 표시 (녹화 방송용) |
| Trustless/Secure Delay | Live Canvas에서 홀카드 숨김 (생방송용) |
| Manual | RFID 없이 수동 카드 입력 |
| Commentary | 해설자 전용 보안 분리 뷰 |

### 특수 규칙

| 규칙 | 설명 |
|------|------|
| Bomb Pot | 모든 플레이어가 동일 금액을 Preflop에 투입, Flop부터 시작 |
| Run It Twice | 올인 후 보드를 2번 전개하여 팟 분배 |
| Rabbit Hunting | 핸드 종료 후 나왔을 보드 카드를 공개 |
| Straddle | BB의 2배를 UTG가 블라인드로 추가 |
| Double Board | 2개의 보드를 동시에 전개 |

## 7. 통계와 분석

### 플레이어 통계

게임이 진행될수록 축적되는 핵심 지표:

| 지표 | 의미 | 예시 |
|------|------|------|
| **VPIP** | 자발적 팟 참여율 | 25% → 4번 중 1번 참여 |
| **PFR** | Preflop Raise 비율 | 18% → 적극적 플레이어 |
| **AF** | 공격성 지수 | 3.0 → 매우 공격적 |
| **WTSD** | Showdown 도달 비율 | 30% → 신중한 플레이어 |
| **WIN** | 승률 | 55% → 평균 이상 |

이 통계는 방송 화면에서 플레이어 정보와 함께 표시되어 시청자에게 "이 플레이어가 어떤 스타일인가"를 한눈에 보여준다.

### 승률 계산

17개 게임별 전용 평가기가 있으며, Monte Carlo 시뮬레이션으로 각 베팅 단계마다 실시간 승률을 계산한다. 10,000회 시뮬레이션을 200ms 이내에 완료해야 한다.

승률은 상황에 따라 극적으로 변한다:

| 상황 | Player A (A♠K♥) | Player B (Q♣Q♦) |
|------|:---------------:|:---------------:|
| Preflop | 43% | 57% |
| Flop: T♣ 8♦ 3♥ | 24% | 76% |
| Turn: A♦ | 89% | 11% |

이 승률 변화가 실시간으로 방송 화면에 반영되어 시청자의 몰입도를 높인다.

### 핸드 히스토리

모든 핸드는 기록되어 사후 분석이 가능하다: 필터링 (게임 타입, 플레이어, 팟 크기), 리플레이 (핸드별 재생), Export (CSV/JSON).

## 8. 데이터 구조

### 왜 기획 단계에서 데이터 구조를 논의하는가

화면에 보이는 모든 요소 — 카드, 칩, 승률, 통계 — 는 데이터에서 온다. 어떤 데이터를 저장하느냐가 "무엇을 보여줄 수 있느냐"를 결정한다.

- PlayerState에 `country` 필드가 없으면 국기를 표시할 수 없다
- TournamentState가 없으면 토너먼트 기능 (남은 인원, 상금)을 넣을 수 없다
- DisplayState가 없으면 Dual Canvas (Live/Delayed 분리 출력)를 구현할 수 없다
- Action 기록이 없으면 핸드 히스토리 리플레이가 불가능하다

**데이터 구조 = 제품의 가능성 범위**

### 핵심 엔티티 관계도

![핵심 엔티티 관계도 (ERD)](images/prd/prd-entity-erd.png)

#### 6개 핵심 엔티티

| 엔티티 | 왜 필요한가 | 핵심 속성 |
|--------|-----------|----------|
| **Event** | "WSOP Main Event 2026" 같은 대회 단위 | 이름, 게임 타입, 블라인드 |
| **GameSession** | 핸드 1회의 모든 정보를 묶는 단위 | 핸드 번호, 게임 타입(22종), 딜러 좌석, 상태 |
| **TableState** | 테이블 중앙 상태 | 메인팟, 사이드팟 배열, 보드 카드 5장, 현재 단계 |
| **PlayerState** | 좌석별 플레이어 정보 (최대 10명) | 이름, 칩, 홀카드(2~7장), 국적, 사진, 상태 |
| **BettingState** | 현재 베팅 라운드 진행 | 현재 베팅액, 최소 레이즈, 차례, 남은 시간 |
| **Action** | 각 베팅 행동 기록 | 액션 타입(7종), 금액, 타임스탬프 |

#### 보조 엔티티

| 엔티티 | 왜 필요한가 | 핵심 속성 |
|--------|-----------|----------|
| **DisplayState** | 화면 표시 제어 | 홀카드 공개 여부(좌석별), Trustless 모드, 지연 시간, 애니메이션 상태 |
| **TournamentState** | 토너먼트 전용 정보 | 참가자 수, 남은 인원, 상금 구조, 레벨 정보 |

#### 관계

- Event → GameSession: 1:N (하나의 이벤트에서 여러 핸드 진행)
- GameSession → TableState: 1:1
- GameSession → PlayerState: 1:N (최대 10명)
- PlayerState → Action: 1:N (한 플레이어의 여러 베팅 행동)

### 설정 구조

시스템 전체 설정은 282개 필드로 구성되며, 이를 11개 도메인별 레코드로 분해하여 관리한다:

ServerConfig, RfidConfig, RenderConfig, NetworkConfig, GameConfig, OutputConfig, SkinConfig, AnalyticsConfig, SecurityConfig, ExternalConfig, UiConfig
