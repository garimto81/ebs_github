## D6 — `/auth/login` envelope vs `/auth/refresh` flat

### 결정: (c) 현 상태 유지

- `/auth/login` 응답: `{ data: {...}, error: null, meta: null }` envelope
- `/auth/refresh` 응답: `{ accessToken, refreshToken, ... }` flat (envelope 없음)
- 시나리오 측에서 endpoint별 분기로 흡수

### 근거

| 사유 | 설명 |
|------|------|
| **영향 최소화** | 시나리오 1줄 분기로 해결. backend 변경 0줄 |
| **Refresh 호출 빈도** | refresh는 access token 만료 직전마다 호출. envelope wrapping 부담을 prod에서 제거 |
| **Login은 사용자 보고용** | error/meta 필드가 사용자 직접 노출 메시지 + audit context. envelope 가치 있음 |
| **Refresh는 client 자동** | accessToken/refreshToken만 필요. 추가 메타데이터 불필요 |

### 시나리오 가이드 (integration-tests)

```
# /auth/login 응답 access path
.data.accessToken
.data.refreshToken
.data.user.userId

# /auth/refresh 응답 access path (flat)
.accessToken
.refreshToken
```

테스트 작성 시 두 endpoint를 분기 처리. 이는 v0.4.3 default이며 별도 변경 안 함.

### 향후 통일 시점

운영에서 envelope 통일 필요성이 발생하면 별도 PR로 cascade. 추적: B-Q21 (envelope unification timeline).

## D7 — single-session refresh policy

### 결정: (a) 의도 — 시나리오 측 격리 가이드만

- 동일 user 새 login 시 이전 refresh token jti 무효화
- 시나리오 10.x sequential 실행에서 10.3(refresh)가 10.2(live login) 후 401 → **의도된 보안 동작**

### 근거

| 사유 | 설명 |
|------|------|
| **보안 우선** | token 탈취 시 invalidation 명확. attacker가 새 login으로 victim refresh token 무효화 가능 = surface narrows |
| **OAuth 2.0 표준** | RFC 6819 §5.2.2.3 권고: refresh token rotation + single-active 정책 |
| **Multi-device 미지원** | EBS Phase 1은 운영자 1 device 1 session 가정. multi-device는 trusted-device grant 별도 cascade (Phase 2 후보) |
| **Audit 명확성** | 1 user = 1 active session → audit log timeline 단순. forensic 시 결정적 |

### 시나리오 격리 가이드

```
# 잘못된 패턴 (10.x sequential)
1. POST /auth/login (live)         → token A
2. POST /auth/refresh (using A)    → token B
3. POST /auth/login (re-login)     → token C  # token B invalidated!
4. POST /auth/refresh (using B)    → 401 (expected)

# 올바른 패턴 (separate session groups)
session_alpha:
  1. POST /auth/login    → A
  2. POST /auth/refresh  → B (A invalidated)
session_bravo:
  1. POST /auth/login    → C (B invalidated)
  2. POST /auth/refresh  → D (C invalidated)
```

각 session group은 독립 user 또는 명시적 logout 후 다음 group 시작. 같은 user 동시 multiple refresh 가정 금지.

### 정책 명시

본 §D7이 BS-01 Authentication.md의 single-session refresh policy를 정식화한다. 향후 `Authentication.md` 차기 cascade에서 본 결정 인용.

## D8 — `POST /skins` JSON-only vs CCR-013 multipart

### 결정: (a) backend `POST /skins/upload` (multipart) 추가

CCR-013 §1 spec 정합 우선. JSON-only metadata는 deprecate path로 유지.

### 작업 (2026-05-10 IMPLEMENTED)

| 변경 | 파일 | 라인 |
|------|------|------|
| `POST /api/v1/skins/upload` multipart endpoint 신규 | `team2-backend/src/routers/skins.py` | + ~110 lines |
| `_validate_gfskin_bytes` helper (tools/validate_gfskin.py 7-stage 재사용) | 동 파일 | + ~50 lines |
| `SKINS_STORAGE_ROOT` env (`EBS_SKINS_STORAGE`, default `/app/data/skins`) | 동 파일 | + 1 const |
| `MAX_GFSKIN_BYTES` 50 MB guard (SG-004 §1) | 동 파일 | + 1 const |

### Endpoint 계약

```
POST /api/v1/skins/upload
Content-Type: multipart/form-data
Authorization: Bearer <admin_token>

field "file": .gfskin (ZIP archive)

→ 201 Created
{
  "data": {
    "skinId": int,
    "name": str,           # manifest.name.en (또는 ko/첫 키)
    "description": str | null,
    "themeData": str,      # JSON: {spec_version, version, rive_*, supported_output_events, storage_*, byte_size}
    "isDefault": bool,
    "createdAt": str,
    "updatedAt": str
  },
  "meta": {
    "manifest": {"skinId": uuid, "version": semver, "specVersion": str, "riveArtboard": str, "events": int},
    "storageId": hex32,
    "sizeBytes": int
  }
}
```

### 에러 응답 (SG-004 코드)

| HTTP | code | 의미 |
|:----:|------|------|
| 400 | GFSKIN_BAD_EXTENSION | 파일명이 .gfskin 미확장 |
| 400 | GFSKIN_EMPTY | 빈 업로드 |
| 413 | GFSKIN_ZIP_TOO_LARGE | 50MB 초과 |
| 422 | GFSKIN_ZIP_TOO_MANY_FILES | 200 파일 초과 |
| 422 | GFSKIN_MANIFEST_MISSING | manifest.json 없음 |
| 422 | GFSKIN_MANIFEST_INVALID | JSON Schema 위반 |
| 422 | GFSKIN_RIVE_MISSING | rive_file 미존재 |
| 422 | GFSKIN_RIVE_BAD_MAGIC | Rive magic bytes 위반 |
| 422 | GFSKIN_PREVIEW_MISSING | preview.png 미존재 |
| 422 | GFSKIN_PREVIEW_BAD_SIZE | 512×288 위반 |
| 422 | GFSKIN_AUDIO_REF_MISSING | audio file 참조 미존재 |
| 422 | GFSKIN_EVENT_UNKNOWN | API-04 OutputEvent 카탈로그 외 이벤트 |
| 422 | GFSKIN_VERSION_MISMATCH | min_ebs_version > current |
| 409 | DUPLICATE | 동일 name skin 이미 존재 (rollback storage) |

### 기존 `POST /skins` (JSON metadata) 처리

- **Deprecation 마크 미부착** (현 단계). v0.5.x 시점에 deprecation header + sunset 6개월 정책 후속 cascade
- 두 endpoint 병존: lightweight metadata-only는 JSON, 실제 .gfskin 배포는 multipart

### Storage 정책

- 경로: `${EBS_SKINS_STORAGE:-/app/data/skins}/<uuid32>.gfskin`
- 크기: 최대 50MB / archive (SG-004 §1)
- 수명: 무한 (skin row 존재 동안). DELETE /skins/{id} 시 storage cleanup은 Phase 2 후보 (현재는 row만 삭제)
