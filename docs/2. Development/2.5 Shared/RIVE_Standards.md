## 한 줄 정의

> **Overlay Graphics** = 시청자가 포커 방송을 보면서 "지금 이 테이블에서 무슨 일이 벌어지고 있는지" 단번에 알 수 있도록, 카메라 영상 위에 실시간으로 덧씌우는 모든 그림과 글자와 숫자.

# Part I — 왜 그리는가

<a id="ch1"></a>

## Ch.1 — 보이지 않는 것을 그리다

축구 중계에서 점수판은 어렵지 않습니다. 점수와 공의 위치는 모두가 보는 **공개 정보** 이기 때문입니다. 카메라가 잡는 그대로를 화면에 정리하면 됩니다.

포커는 다릅니다. 시청자가 가장 궁금해하는 정보가 **두 겹으로 가려져** 있습니다 (Foundation.md §Ch.1.1).

| 구분 | 축구 / 야구 | 포커 |
| --- | --- | --- |
| 카드 / 패 | 해당 없음 | **비공개** (뒤집힘) |
| 선수 액션 | **공개** (눈으로 명확) | **모호** (칩 동작 작고 빠름) |
| 그래픽 역할 | 보이는 정보 **정리** | 보이지 않는 정보 **생성** |

> 포커는 카드뿐 아니라 액션도 별도 그래픽으로 알려줘야 하는 **유일한 메이저 스포츠** 입니다 (Foundation.md §Ch.1.1).

**Overlay Graphics 는 이 두 겹의 비대칭을 풀어 시청자에게 건네주는 마지막 손길** 입니다. EBS 시스템 전체가 — RFID 안테나, Command Center, Game Engine, Render Pipeline — 결국 이 한 장의 화면을 그리기 위해 존재합니다.

```
   +-------------------------------------------------------+
   |                                                       |
   |     +----------------------+                          |
   |     | (포커 테이블 영상)    |                          |
   |     |                      |    [없으면 시청자는      |
   |     |                      |     아무것도 모릅니다.]   |
   |     +----------------------+                          |
   |                                                       |
   |     ↓ Overlay 가 덧씌워지면                            |
   |                                                       |
   |     +----------------------+                          |
   |     | [홀카드: A♠ K♠]      |                          |
   |     | [팟: 12,500]         |    [시청자가 비로소       |
   |     | [현재 액션: RAISE]    |     게임을 이해합니다.]   |
   |     +----------------------+                          |
   |                                                       |
   +-------------------------------------------------------+
              ASCII MOCK · overlay 의 효과 (전/후)
```

## Ch.2 — 1 초의 약속

방송 그래픽에는 두 종류가 있습니다.

| 종류 | 예시 | 허용 지연 |
|------|------|:--------:|
| **Pre-rendered** | 영화 자막, 광고 인서트 | 수 초 ~ 수 분 |
| **Real-time** | Overlay Graphics, 스포츠 점수판 | **100ms 이내** |

운영자가 Command Center 에서 `RAISE 5,000` 버튼을 누르는 순간, 시청자 화면에 그 액션이 뜨기까지 **1 초를 넘기면 안 됩니다**.

| 시청자 인지 | 누적 지연 |
|------------|:--------:|
| "라이브답다" | < 500ms |
| "방송이 좀 늦네" | 500ms ~ 1.5s |
| "이거 라이브 맞아?" | > 2s |

**Overlay Graphics 의 첫째 약속은 정확성, 둘째 약속은 속도** 입니다.

```
  운영자 클릭          Engine 처리       Render 출력      시청자 인지
   |                   |                |                |
   t=0ms ----- 30ms ---- 50ms ---- 80ms ---- ~~ ---- 100ms (목표)
   |                   |                |                |
  [BTN]              [Event]         [Frame N]        [eye blink]

  허용 budget: 100ms
```

## Ch.3 — 무대와 막의 비유

이 문서 전체는 **연극 무대** 의 비유로 통일됩니다. Foundation.md 가 EBS 를 "포커 방송의 마법사" 로, Lobby_PRD.md 가 Lobby 를 "관제탑" 으로, Command_Center_PRD.md 가 CC 를 "조종석" 으로 표현했다면 — Overlay 는 **그 조종이 만들어내는 무대** 입니다.

| 비유 | Overlay 시스템 | 다른 PRD 와의 위치 |
|------|---------------|------------------|
| **무대** (stage) | 방송 화면 (1920 × 1080) | Lobby = 관제탑 / CC = 조종석 |
| **배경막** (backdrop) | 카메라 영상 | (외부 입력) |
| **막** (curtain layer) | Z-layer (9 단) | Ch.21 |
| **배우** (actor) | 그래픽 요소 (Player Card 등) | Ch.9 ~ Ch.20 |
| **연기** (acting) | Animation (Rive state machine) | Ch.24 |
| **연출가** (director) | Game Engine | (외부 시스템) |
| **무대감독** (stage manager) | Render Pipeline | (외부 시스템) |
| **각본** (script) | OutputEvent (21 종) | RIVE_Event_Protocol.md |
| **관객** (audience) | 시청자 | — |
| **백스테이지** | 운영자 화면 (debug 영역) | Ch.27 |

이 비유를 머릿속에 두면 — 기술 명세 없이도 Overlay 의 거의 모든 결정 이유가 자연스럽게 이해됩니다.

## Ch.4 — 1 단계: 대회 등록

Overlay 가 첫 픽셀을 그리려면 — 어떤 대회를 송출할지가 정해져야 합니다.

### 4.1 Tournament 카드 한 장

```
  +======================================================+
  |   대회 등록 (Tournament Setup)                        |
  +======================================================+
  |   대회명:  [ ___________________________ ]            |
  |   날짜:    [ YYYY-MM-DD ~ YYYY-MM-DD     ]            |
  |   장소:    [ ___________________________ ]            |
  |   상금:    [ Total Prize Pool             ]            |
  |   바이인:  [ ____________                 ]            |
  |   레벨:    [ 분 / 레벨                    ]            |
  |   브랜딩:  [ {tournament-id}.brand        ]            |
  |                                                      |
  |   [등록 → 다음 단계]                                  |
  +======================================================+
              ASCII MOCK · 대회 등록 카드
```

<!-- IMG_TODO: Lobby 의 대회 등록 화면 실제 스크린샷 (Lobby_PRD §5 참조) -->

이 한 장에 입력된 정보는 시청자 화면 모서리의 로고, 좌측 하단 레벨 표시, 상단의 남은 시간, 그리고 모든 그래픽의 **컬러 팔레트** 까지 결정합니다.

### 4.2 브랜딩 패키지

`{tournament-id}.brand` 라는 한 파일은 **그래픽 의상 한 벌** 입니다.

| 의상 항목 | 내용 |
|-----------|------|
| 컬러 팔레트 | Primary / Secondary / Accent / 텍스트 |
| 폰트 | 영문 / 한글 / 숫자 전용 |
| 로고 (3 종) | 메인 / 워터마크 / 코너 |
| 그래픽 모티프 | 배경 패턴 |
| 사운드 (선택) | 액션 효과음 |

> **외부 출처**: 브랜딩 패키지 스키마는 Foundation.md §Ch.4.2 (개발 대상 오버뷰) 의 "Render & Hardware" 섹션 참조. `.brand` 파일 형식은 1 단계 설계상 가설이며, 실제 형식은 외부 개발팀 합의로 확정됩니다.

## Ch.6 — 3 단계: 운영자 할당과 게임 설정

### 6.1 운영자 = Operator

테이블 1 개 = 운영자 1 명. 큰 결승은 백업 운영자 1 명을 추가 (Command_Center_PRD.md §Ch.5 참조).

| 역할 | 책임 | 도구 |
|------|------|------|
| **Operator** | 한 테이블 12 시간 송출 | Command Center |
| **Backup Operator** (선택) | 메인 운영자 교대 시 | CC (read-only) |
| **Producer** | 여러 테이블 모니터링 | Lobby |

## Ch.7 — 4 단계: 입력 채널 깨우기

### 7.1 입력의 3 줄기

```
   +---------------------+
   |   1. RFID 카드      | -> 어떤 카드가 어디에  (자동, 1단계 핵심)
   +---------------------+
   |   2. Command Center | -> 어떤 액션이 일어났는가 (운영자 키 입력)
   +---------------------+
   |   3. Vision (2단계) | -> 칩 스택 / 좌석 자동 인식 (미래)
   +---------------------+
            |
            v
       [Game Engine]
            |
            v
       [Overlay Graphics]
```

> **단계 구분 (Foundation.md §Ch.1.6)**: 1 단계 = RFID + CC. 2 단계 = + Vision Layer (Foundation.md §Ch.7.3).

### 7.2 RFID 안테나

총 **11 개** 안테나: 좌석 10 + 보드(커뮤니티) 1.

```
  +----------------------------------------------+
  |                                              |
  |          [Comm Card Antenna]                 |
  |              · · · · ·                       |
  |          (테이블 중앙)                        |
  |                                              |
  |   [P1] [P2] [P3] [P4] [P5]                   |
  |    안테나 1~5 (왼쪽 5 좌석)                   |
  |                                              |
```

## Ch.11 — 카드: 홀카드 + 커뮤니티

포커 그래픽의 **핵심** 입니다.

### 11.1 홀카드

```
   카메라 화면 (실물)             Overlay 가 덧씌운 화면

   +-------+                       +-------+
   | (뒤)  |                       | A♠ K♠ |
   |       |       =>              | (앞면 |
   |       |                       |  공개) |
   +-------+                       +-------+
   플레이어 손에 있는              운영자만 본 카드를
   뒤집힌 카드                      Overlay 가 시청자에게 공개
```

원리:

1. RFID 가 카드 NFC 를 읽어 무늬 + 숫자 인식 (Foundation.md §Ch.1.3).
2. Engine 이 OutputEvent 발행 (예: `HOLE_CARD_REVEALED { seat: 3, cards: ["AS", "KS"] }`).
3. Render 가 P3 의 카드 위치에 SVG 덧씌움.
4. 단 — 이 그래픽은 **방송 송출 채널 (PGM = Program, 시청자가 보는 채널) 에만** 표시. 테이블 디스플레이에는 안 됨 (Ch.27 채널 분리 정본).

## Ch.21 — 9 단 z-layer

### 21.1 9 단 구조

```
  z = 0   [카메라 영상 (배경막)]              <- 가장 뒤
  z = 1   [테이블 boundary 표시]
  z = 2   [Player Card 배경 박스]
  z = 3   [카드 (홀카드 + 커뮤니티)]
  z = 4   [칩 스택 + 베팅 라인]
  z = 5   [팟 박스]
  z = 6   [블라인드 / 레벨 / 시계]
  z = 7   [브랜딩 (로고, 워터마크)]
  z = 8   [액션 알림 / ALL-IN 박스]
  z = 9   [모달 / 우승 화면 / 시스템 알림]    <- 가장 앞
```

### 21.2 왜 9 단인가

| 기준 | 이유 |
|------|------|
| 시각 위계 | 시청자 시선 흐름 (배경→콘텐츠→알림) |
| z-fighting 방지 | 같은 z 의 그래픽 깜빡임 방지 |
| 차폐 (occlusion) | 모달 등장 시 그 아래 흐림 |
| 성능 | GPU layer 별 culling + caching |

> 9 단은 본 문서의 1 차 제안 — 외부 개발팀이 GPU 성능 / 디자이너 워크플로우에 따라 8 단 또는 10 단으로 조정 가능.

### 21.3 차폐의 사례

```
  ALL-IN 발생 시:
  z = 0 ~ 7  -> blur (배경 흐림)
  z = 8      -> ALL-IN 박스 (선명)
  z = 9      -> 시스템 알림 (있다면 더 위)
```

## Ch.24 — Rive 와 24 프레임

### 24.1 왜 Rive 인가

| 비교 | After Effects + 영상 출력 | Rive |
|------|--------------------------|------|
| 파일 형식 | MP4 / MOV | `.riv` (벡터 + 상태 머신) |
| 크기 | 큼 | 작음 |
| 동적 변경 | 불가 (재렌더링 필요) | **변수 주입** |
| 상태 전이 | 불가 | **State Machine** |
| 인터랙티브 | 불가 | **이벤트 hook** |
| GPU | CPU 디코딩 | **GPU 직접 렌더링** |

> 정량 비교는 외부 벤치마크에서 확정 — 본 표는 정성 비교.

### 24.2 State Machine 예시

```
   PlayerCard.riv 의 상태 머신
   +--------------------------------------------+
   |                                            |
   |   [hidden]                                 |
   |      |                                     |
   |   onEntry                                  |
   |      v                                     |
   |   [entering] (페이드 인 + 슬라이드)          |
   |      |                                     |
   |   onActionComplete                         |
   |      v                                     |
   |   [idle] ---- onAction --> [highlighted]   |
   |      ^                          |          |
   |      +--- timeout 또는 ----------+          |
   |           onActionComplete                 |
   |      |                                     |
   |   onFold                                   |
   |      v                                     |
   |   [folded] (회색 + dim)                     |
   |      |                                     |
   |   onExit                                   |
   |      v                                     |
   |   [exiting] (페이드 아웃)                    |
   +--------------------------------------------+
```

OutputEvent 가 trigger (예: `PLAYER_FOLD` → `onFold`). 정본은 RIVE_Event_Protocol.md.

### 24.3 프레임 레이트

| 옵션 | 의미 |
|------|------|
| 24 fps | 영화 표준 |
| 30 fps | 일반 TV |
| 60 fps | 게임 / 빠른 스포츠 |

> 본 문서는 **30 또는 60 fps** 를 권장하며 (실시간 액션 매칭), 24 fps 는 영화적 효과 시 옵션. 최종 결정은 외부 방송 PD + 디자이너.

## Ch.25 — 게임 흐름과 그래픽 전이

### 25.1 5 단계 (Hold'em 기준)

```
  +-------------+    +-------------+    +-------------+
  |  Preflop    |    |    Flop     |    |    Turn     |
  | (홀카드만)   | -> | (커뮤니티 3) | -> | (커뮤니티 4) | ->
  +-------------+    +-------------+    +-------------+

  +-------------+    +-------------+
  |   River     |    |  Showdown   |

```

```
              |
              v
  모든 베팅 라인 -> 팟 sweep (~0.8s)
              |
              v
  팟 카운트업 tween (~0.5s)
              |
              v
  다음 단계 (Flop / Turn / River) 카드 등장
```

## Ch.27 — 운영자 시점 vs 시청자 시점

같은 시간 두 화면이 동시에 그려집니다.

### 27.1 두 화면의 차이

```
  +---------------------------------------+        +---------------------------------------+
  |   시청자 화면 (PGM)                    |        |   운영자 화면 (OPS)                    |
  +---------------------------------------+        +---------------------------------------+
  |                                       |        |                                       |
  |   (모든 그래픽 — 깨끗하게)             |        |   (PGM 과 동일)                        |
  |                                       |        |                                       |
  |                                       |        |   ----------------------------         |
  |                                       |        |                                       |
  |                                       |        |   RFID Status: ...                    |
  |                                       |        |   Engine Lag: ...                     |
  |                                       |        |   Last Event: ...                     |
  |                                       |        |                                       |
  +---------------------------------------+        +---------------------------------------+
```

### 27.2 OPS 전용 정보

| 정보 | 의미 | 위치 |
|------|------|------|
| RFID 안테나 상태 | 11 안테나 OK/경고/단절 | 우하단 |
| Engine 지연 | OutputEvent 발행 | 우하단 |
| Render 지연 | 프레임 출력 | 우하단 |
| Last Action 로그 | 최근 N 액션 | 좌하단 |
| Mock 모드 표시 | 시뮬레이션 모드일 때 | 좌상단 빨간 배지 |
| Safe Zone 가이드 | Title/Action Safe 라인 (토글) | 화면 전체 |

### 27.3 채널 분리 보장

| 채널 | 송출 |
|------|------|
| PGM | 시청자용. OPS 정보 절대 송출 안 됨 |
| OPS | 운영자 모니터만 |
| BAK | PGM 동일 (장애 대비) |
| REC | PGM 동일 (아카이브) |

> **하드웨어 SDI 라우터** 에서 분리 보장. 소프트웨어 토글 단독은 부족.

<a id="ch28"></a>

## Ch.28 — 장애 / 누락 / 복구

### 28.1 4 종 시나리오

| 장애 | 증상 | 복구 |
|------|------|------|
| RFID 단절 | 카드 인식 멈춤 | OPS `MANUAL CARD INPUT` 모드 (Command_Center_PRD §Ch.7 Mock vs Real) |
| Engine 응답 없음 | 액션 그래픽 멈춤 | Render 마지막 상태 유지 + OPS 빨간 알림 + 자동 재연결 시도 (timeout 임계값 1차 가설: heartbeat 3 회 연속 실패) |
| Render 장애 | 화면 freeze | BAK 채널 자동 fallback (감지 임계값 1차 가설: 프레임 출력 ~1s 부재) |
| 카메라 피드 끊김 | 배경 검은 화면 | Overlay 솔리드 배경 위에 표시 유지 |

### 28.2 OPS 알림

```
  +----------------------------------------+
  |  RFID Antenna 7 disconnected            |
  |  Last seen: {N} seconds ago             |
  |  [Reconnect] [Mock Mode] [Ignore]       |
  +----------------------------------------+
              ASCII MOCK · OPS 알림 토스트
```

시청자 화면에는 장애 보이지 않음 — 마지막 상태 유지 또는 BAK fallback.

### 28.3 복구 우선순위

```
  1. 시청자 화면 끊김 방지 (BAK fallback)
  2. 운영자에게 즉시 알림 (OPS 토스트)
  3. 자동 재연결 시도
  4. 수동 복구 (운영자 개입)
  5. 사후 로그 (장애 기록 -> 다음 방송 개선)
```

# Part VII — 진화

## Ch.30 — 1 단계 → 2 단계 그래픽 변화

EBS 는 **2 단 진화** 로 설계 (Foundation.md §Ch.1.6, §Ch.7.3, §Ch.9.2).

### 30.1 두 단계 정의

| 단계 | 입력 | 운영자 |
|------|------|--------|

| 2 단계 (미래) | RFID + Vision Layer + AI | 모니터링만 |

### 30.2 그래픽 변화의 핵심

**시청자 화면 (PGM) 은 거의 동일** — 차이는 **백스테이지** 에 있습니다.

| 요소 | 1 단계 | 2 단계 | 시청자 차이 |
|------|--------|--------|:--------:|
| PGM | 모든 그래픽 | 동일 | 없음 |
| OPS | Debug + 컨트롤 | 모니터링 + 정정 | (시청자엔 안 보임) |
| Player Card | RFID + 좌석 매핑 | + Vision 자동 좌석 | 없음 |
| 액션 표시 | CC 키 입력 → 그래픽 | Vision AI → 그래픽 | 없음 |
| 칩 스택 | Engine 계산 | Vision 자동 인식 | 정확도 ↑ |

### 30.3 OPS 화면 단순화

```
  1 단계 OPS:                       2 단계 OPS:
  +------------------------+        +------------------------+
  | [PGM 화면]              |        | [PGM 화면]              |
  |                        |        |                        |
  | [8 액션 버튼]           |        | [Monitor only]          |
  | [Stack input]           |        | [AI confidence]         |
  | [Card override]         |        | [Override 버튼 소수]     |
  | [Debug 영역]            |        | [Debug 영역]            |
  |                        |        |                        |
  +------------------------+        +------------------------+
       12시간 입력 노동              12시간 모니터링 + 가끔 정정
```

### 30.4 한 마디 요약

> 그래픽이 변하는 게 아니라 — **그래픽을 만드는 사람** 이 변함. 1 단계 = 운영자, 2 단계 = AI.

### 31.2 FAQ

**Q1. Overlay 가 안 보이면 시청자는?**
카메라 영상만 봅니다 — 카드도 액션도 모름. 그래서 필수.

**Q2. 운영자 1 명이 12 시간 한 테이블만?**
네 (Command_Center_PRD §Ch.5). 결승은 백업 1 명 추가.

**Q3. Mixed Game 그래픽 변화?**
게임 전환 ~5s 카운트다운 + 종목 라벨 + 카드 영역 레이아웃 변경 (Game_Rules/ 정본).

**Q4. 모바일에서도 보이나?**
네. 모든 그래픽 Title Safe 90% 안. OPS 정보는 모바일에 안 보임.

**Q5. AR/VR 지원?**
1 단계에는 미지원. 2 단계 이후 검토.
