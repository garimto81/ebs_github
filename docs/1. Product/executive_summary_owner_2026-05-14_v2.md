# EBS 오너 Executive Summary

> *9 슬라이드. 한 슬라이드에 한 메시지.*
> *세 가지 결정을 받기 위해 만든 자료.*

---

## Slide 1 / Why EBS — 한 문장으로

**포커 방송에서 시청자가 가장 보고 싶은 두 가지 정보(누구의 카드인가, 얼마를 베팅했는가)를 실시간으로 화면에 띄워주는 자체 개발 라이브 그래픽 시스템.**

축구 중계는 점수판이 있으면 끝난다. 포커는 매 초마다 점수판을 새로 만들어야 한다. 그 점수판 공장이 EBS다.

```
   ┌───────────────────────────────────────────────┐
   │   카메라 원본 영상 (포커 테이블)               │
   │                                               │
   │       ┌──[좌석 3]──┐                          │
   │       │  ♠A  ♥K   │ ← 홀카드 자동 인식 + 표시 │
   │       │  스택 50K  │                          │
   │       └────────────┘                          │
   │                                               │
   │           ┌─[중앙 팟]──┐                      │
   │           │  POT 120K  │ ← 베팅 토큰 합산     │
   │           └────────────┘                      │
   └───────────────────────────────────────────────┘
   카메라 영상 위에 9 종류 그래픽이 실시간으로 덧씌워진다.
```

> *EBS = 실시간 정보 생성기. 후편집 그래픽(리더보드·자막틀)은 EBS 영역 아님.*

## Slide 2 / 왜 지금 — 시장 맥락

라이브 포커 방송 시장은 두 가지 변화가 동시에 진행 중이다.

| 변화 | 의미 |
|------|------|
| **기존 솔루션의 한계가 드러남** | 매년 라이선스 + 외부 의존 + 입력 모델이 옛날 그대로 |
| **자체 개발 스택이 성숙함** | 화면(Flutter) + 애니메이션(Rive) + 자체 카드 인식 하드웨어 = 통합 가능 시점 |

기존 솔루션은 *운영자가 매 베팅을 직접 키보드로 입력*하는 방식에서 진화하지 않았다. EBS는 그 방식을 1단계로 그대로 채택하되, 2단계에서 **카메라 6대 + 컴퓨터 비전**으로 사람을 빼는 길을 처음부터 설계에 반영했다.

```
   기존 솔루션              →            EBS 자체 개발
   ─────────                            ──────────────
   매년 라이선스 비용       →            1회성 투자 + 영구 자산
   외부 의존                →            데이터 100% 통제
   영구 수동 입력           →            1단계 수동 → 2단계 무인화
   1차원 그래픽 (홀덤 위주) →            9 카테고리 × 22 게임 통합
```

> *지금이 자체 스택으로 통합 가능한 첫 시점.*

## Slide 3 / 핵심 차별점 3가지

기존 솔루션 대비 EBS만의 세 가지.

| # | 차별점 | 한 줄 |
|:-:|--------|------|
| ① | **영구 자산** | 매년 라이선스가 아니라 한 번 만들면 영구 사용 + 자유 확장 |
| ② | **22 게임 × 9 그래픽 통합** | 홀덤만이 아닌 오마하 / 스터드 / 드로우 / 믹스드 게임 22종을 같은 시스템에서 운용 |
| ③ | **무인화 로드맵** | 1단계 운영자 수동 → 2단계 카메라 + 컴퓨터 비전 자동 인식 (사람 없는 라이브 송출) |

```
   ┌─ ① 영구 자산 ──┐  ┌─ ② 통합 깊이 ─┐  ┌─ ③ 미래 진화 ──┐
   │                │  │                │  │                 │
   │   매년 ↓       │  │   게임 22종    │  │   사람 운영 →   │
   │   영구 ↑       │  │   그래픽 9종   │  │   자동 인식     │
   │                │  │                │  │                 │
   └────────────────┘  └────────────────┘  └─────────────────┘
       비용 구조          제품 범위          입력 방식
```

> *세 차별점은 독립적이지 않다. ① 자산이 있어야 ② 통합이 가능하고, ② 통합이 있어야 ③ 무인화 학습 데이터가 쌓인다.*

## Slide 4 / 제품 구성 — 4 컴포넌트

EBS는 더블 클릭으로 실행되는 한 개 프로그램이 아니라, 톱니바퀴처럼 맞물린 네 개 컴포넌트다.

| 컴포넌트 | 한 줄 역할 | 누가 보는가 |
|----------|-----------|-----------|
| **Lobby** | 시프트 시작 5분 안에 룰·좌석·카드 인식 검증을 끝내는 게이트웨이 | 매니저 |
| **Command Center** | 본방송 중 운영자 시선이 85% 머무는 1×10 좌석 조종석 (6 키) | 컨트롤룸 운영자 |
| **Game Engine** | 22 게임 룰 + 21 출력 이벤트를 처리하는 두뇌 (사람이 직접 보지 않음) | 시스템 내부 |
| **Back Office** | 외부 대회 시스템 동기화 + 권한 검증 + 모든 핸드 데이터 보관소 | 어드민 |

```
   사용자 시점 흐름
   ─────────────

   [Lobby] ─── 클릭 ───> [Command Center] ─── 키 입력 ───>
   매니저 5분             운영자 본방송 중             │
   사전 검증              실시간 액션 입력             │
                                                       v
                                                  [Game Engine]
                                                   22 룰 처리
                                                       │
                                                       v
                                                   [Overlay]
                                                  방송 화면 출력
        모든 데이터 흐름은
        [Back Office] ──── DB + 권한 + JSON 데이터 공급
        를 통해 동기화
```

> *Game Engine과 Overlay는 사용자가 직접 만지지 않지만 가장 일을 많이 한다.*

## Slide 5 / 기술 스택 한 줄

**화면(Flutter) + 애니메이션(Rive) + 자체 RFID 카드 인식.** 끝.

```
   ┌─────────────────────────────────────────┐
   │                                         │
   │     Flutter            Rive             │
   │     (화면 UI)          (그래픽 애니)    │
   │         +                +              │
   │              자체 RFID 카드 인식         │
   │                (카드 한 장 한 장 식별)   │
   │                                         │
   └─────────────────────────────────────────┘

   왜 이 조합?
   ─────────
   - 같은 코드로 PC + 웹 동시 지원 (Flutter)
   - 디자이너가 만든 애니를 그대로 사용 (Rive)
   - 외부 업체에 의존하지 않는 카드 인식 (자체 RFID)
```

> *디테일(언어 / 프레임워크 버전 / 통신 프로토콜)은 개발팀 인계 PRD 별도 참조.*

## Slide 6 / 진행 현황 — 2026-05-14 기준

지난 1주일간 사이클 16회 완료. 핵심 검증 모두 통과.

```
   진행 현황 막대
   ─────────────

   개발 사이클        ████████████████░░░░░░  Cycle 16 완료
   v01 통합 테스트    ██████████ PASS  (멀티 핸드)
   v02 통합 테스트    ██████████ PASS  (멀티 핸드 + 깊이)
   v03 통합 테스트    ██████████ PASS  (스트래들 + 앤티 + 분할)
   7일 KPI #215      ██████████ CLOSED (Cycle 7 시점)
   9 카테고리 SSOT   ██████████ v1.0.0 확정 (Cycle 16)
```

| 마일스톤 | 상태 | 의미 |
|----------|:----:|------|
| Cycle 16 완료 | ✅ | Wave 1+2 병렬 dispatch 첫 성공. 작업 속도 약 4배 가속 |
| v01~v03 e2e 모두 PASS | ✅ | 멀티 핸드 + 깊이 + 특수 룰(스트래들·앤티)까지 자동 검증 |
| 7일 KPI CLOSED | ✅ | 첫 주 목표 일정 내 종료 |
| 9 카테고리 SSOT | ✅ | 시청자 화면에 띄울 그래픽 9 종 확정 (5개 PRD 정합 완료) |

> *"통합 테스트 PASS" = 사람 손 안 거치고 자동 스크립트가 22 게임 시나리오를 끝까지 돌려도 화면이 깨지지 않음.*

## Slide 7 / 다음 마일스톤 — 결정 대기 5개

다음 단계로 가기 위해 오너 결정이 필요한 5가지. 내부 라벨로 **B-Q5 ~ B-Q9**.

| 라벨 | 결정 항목 | 묻는 것 |
|------|----------|---------|
| B-Q5 | **MVP 범위** | 22 게임 중 어느 게임부터? 9 그래픽 중 어느 것부터? |
| B-Q6 | **런칭 일정** | 첫 라이브 송출 목표 시즌은? |
| B-Q7 | **업체 선정** | 카드 + 리더 통합 공급 파트너 어느 곳? |
| B-Q8 | **거버넌스** | 운영 책임 / 권한 분리 / 시프트 교대 모델 |
| B-Q9 | **품질 기준** | 얼마나 정확해야 라이브 송출 OK인가? (휴먼 인터벤션 허용 %) |

```
   현재                       미정 5개                       런칭
   ────                       ───────                       ────
    │                                                          ▲
    │                                                          │
    ├──[B-Q5  MVP 범위]──────────────────────────┐             │
    │                                            │             │
    ├──[B-Q6  런칭 일정]─────────────────────────┤             │
    │                                            │  결정 →     │
    ├──[B-Q7  업체 선정]─────────────────────────┼──────────── │
    │                                            │  잠금 해제  │
    ├──[B-Q8  거버넌스]──────────────────────────┤             │
    │                                            │             │
    └──[B-Q9  품질 기준]─────────────────────────┘             │
```

> *이 5개가 잠금 상태인 동안에는 기획서 보강 + 검증 작업만 진행 가능. 출시 백로그로 변환할 수 없음.*

## Slide 8 / 의사결정 요청 3개 (오늘 받고 싶은 것)

5개 미정 항목을 풀기 위해, 오늘 미팅에서 받고 싶은 **세 가지 답**.

| # | 결정 영역 | 구체적 질문 | 받는 형태 |
|:-:|-----------|-------------|----------|
| ① | **일정** | 첫 라이브 송출 목표일은 언제? | 분기 단위 OK (예: 2027 Q2) |
| ② | **MVP 범위** | 22 게임 중 어느 종목부터 시작? 9 그래픽 중 우선순위 3개? | 게임명 1~2개 + 그래픽 카테고리 # |
| ③ | **예산 + 팀 규모** | 자체 개발 5팀 체제 유지? 외부 협력 비율? | 인원수 / 외주 비율 % |

```
   ┌────────────────────────────────────────────┐
   │                                            │
   │   ① 일정    — 언제 라이브 첫 송출?          │
   │                                            │
   │   ② MVP     — 어느 게임 + 어느 그래픽부터?  │
   │                                            │
   │   ③ 예산    — 자체 5팀 + 외부 비율?         │
   │                                            │
   └────────────────────────────────────────────┘

   답 3개 → B-Q5~Q9 잠금 해제 → 출시 백로그 변환
   (오늘 결정 → 1주일 안에 백로그 + 일정표로 변환)
```

> *답이 즉시 안 나와도 OK. "이 조건이 충족되면 ?"" 형태의 조건부 답변도 진행 가능.*

## Slide 9 / Call to Action — 한 줄 마무리

**세 가지 결정만 주시면, 기획서를 출시 일정으로 즉시 변환합니다.**

```
   ┌─────────────────────────────────────────────┐
   │                                             │
   │    결정 3개  →  백로그 변환  →  라이브 송출  │
   │                                             │
   │     오늘     →   1주일 안     →   목표 시즌  │
   │                                             │
   └─────────────────────────────────────────────┘

   다음 단계 (오너 결정 후 자동 트리거)
   ─────────────────────────────────
   1. B-Q5~Q9 잠금 해제 (백로그 그대로 진행)
   2. MVP 범위 = 출시 백로그 변환
   3. 일정 = Wave 단위 마일스톤 자동 생성
   4. 예산 = 팀 배치도 확정
```

> *문서는 충분히 두꺼워졌다. 이제 결정이 필요한 시점.*
