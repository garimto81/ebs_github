## Architecture

EBS Core는 3 입력 → 실시간 오버레이 출력으로 정의된다:

```
WSOP LIVE(대회정보) + RFID(카드) + Command Center(액션) → Game Engine → Overlay Graphics
```

- **실시간 라이브 시스템** (경쟁: PokerGFX)
- **시간축 경계**: EBS = 실시간 라이브, 포스트프로덕션 ≠ EBS (경쟁: Adobe)
- **API 계층**: WSOP LIVE→API→EBS+소비자 / EBS→API→소비자 (단방향)
- **기술 스택**: Flutter Desktop + Rive (Lobby/Settings/GE/CC 공통 스택, Foundation §5.1 — γ 하이브리드 2026-04-22: Lobby Web 정규 배포 + Desktop 개발 디버깅), FastAPI Backend, Pure Dart Game Engine

상세는 [Foundation.md](Foundation.md) "Architecture" 섹션 참조.

### 2 렌즈: 기능 6 ↔ 설치 4 (Foundation §4.4)

시스템은 **기능 6개** 로 구성되나, 실제 **설치 단위** 는 5개 (4 SW + 1 HW, γ 하이브리드 2026-04-22). 개발팀 오독 방지.

| 설치 단위 | 포함 기능 | 언어/스택 | 배포 타겟 · 프로세스 모델 |
|-----------|-----------|-----------|--------------|
| **EBS Lobby Web** | 로비 + 세팅 + Rive 파일 관리 (D3 회의 2026-04-22, 구 Graphic Editor 축소) | Flutter Web + Rive | **Web (Docker nginx 정규)** · Desktop 빌드는 개발자 디버깅 |
| **EBS Desktop App** (CC+Overlay 공동) | 커맨드 센터 + 오버레이 뷰 | Flutter Desktop + Rive | **Desktop 설치** · Foundation §5.0 2 런타임 모드 (탭/슬라이딩 OR 다중창) |
| **Game Engine** | 게임 엔진 | Pure Dart | 별도 서비스 (Docker / `dart run bin/harness.dart`) |
| **Backend Server (BO)** | 백오피스 서버 | FastAPI + SQLite/PostgreSQL | 별도 서비스 |
| **RFID Hardware** | 카드 인식 HW | ST25R3911B + ESP32 | 물리 장비 (USB) |

**팀 분할 vs 설치 단위**:

| 팀 | 소유 기능 | 빌드 산출물 |
|----|-----------|-------------|
| team1 | Lobby + Settings + Rive 파일 관리 | `EBS Lobby Web` (단독) |
| team2 | Backend Server (BO) | `Backend Server` |
| team3 | Game Engine | `Game Engine` |
| team4 | Command Center + Overlay | `EBS Desktop App` (CC + Overlay 공동) |

> **γ 하이브리드 (2026-04-22)**: team1 = Web 정규 배포, team4 = Desktop 설치. 같은 Flutter 스택 공유하지만 배포 타겟 직교. "team1+team4 공동 바이너리" 는 2026-04-21 과도기 표현으로 더 이상 유효하지 않음.
>
> **D3 회의 결정 (2026-04-22)**: 별도 Graphic Editor 앱 제거. Lobby Web 내부에 **Rive 파일 관리 섹션 (업로드 + 검증 + 활성화)** 으로 축소. 메타데이터는 Rive 파일에 내장. 상세: `docs/4. Operations/Critic_Reports/Meeting_Analysis_2026_04_22.md §3 D3`.

### 팀 소유 경계

**코드**:
- 각 팀은 `team{N}-*/` 아래만 수정 가능
- 다른 팀 코드 폴더 접근 금지

**문서** (v7 `free_write_with_decision_owner`, 2026-04-17 CCR 완전 폐기):
- 모든 세션이 모든 docs 를 **자유 편집** 가능 (쓰기 차단 없음)
- 의미적 충돌 발생 시 `team-policy.json` `teams[*].owns` + `contract_ownership[*].publisher` 의 **decision_owner** 가 최종 판정
- 각 팀은 `docs/2. Development/2.{1..4} {팀}/` 가 자기 책임 영역 (decision_owner)
- Shared (`2.5`), Product (`1.`), Operations (`4.`) 는 Conductor 가 decision_owner
- 다른 팀 영역 편집 시 additive 원칙 + 커밋 메시지 notify + decision_owner 최종 판정
- `docs/3. Change Requests/` 는 역사 폴더 (활성 프로세스 아님)

## Communication Rules (CRITICAL)

외부(업체/파트너) 대상 이메일·문서·채팅에 적용되는 규칙.

### 핵심 규칙

| 규칙 | 내용 |
|------|------|
| 회사명 노출 금지 | "BRACELET STUDIO" 외부 이메일에 절대 사용 안 함 |
| 기술 스펙 노출 금지 | 주파수, 프로토콜, IC 칩명 언급 안 함 |
| 서명 | 이름만, 회사명 없음 |
| 이메일 구조 | 상대방 제품 정보만 요청하는 형태 |

### 용어 규칙

| 금지 | 사용 | 이유 |
|------|------|------|
| chips (카지노 맥락) | 베팅 토큰 (betting tokens) | 반도체 칩과 혼동 |
| chips (반도체 맥락) | IC, 반도체 | 위와 구분 |

### 자동화 규칙 (Slack/Gmail)

| 규칙 | 내용 |
|------|------|
| `--notify` 사용 금지 | `chat:write:bot` scope 없음 |
| `--post`만 사용 | 채널 메시지 갱신만 가능 |
| DM 발송 안 함 | Slack API 제한 |
