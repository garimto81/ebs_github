# Critic — SG-008-b11 v1.3 (2026-05-03 사용자 비판)

## TL;DR

| 영역 | v1.3 spec | 사용자 의도 | 정합? |
|------|-----------|------------|:-----:|
| Launch UX | "Web variant 복원, cc_url 반환, browser navigate" | "interaction in lobby — lobby 가 CC 가 active 임을 시각 표현" | ❌ |
| Stand-alone CC re-entry | "manual login form (4 field)" | "비번만, 나머지는 lobby settings" | ❌ |
| Implementation evidence | "9/9 backend E2E PASS" | "deployed frontend 실제 동작" | ⚠ verification gap |

→ v1.3 은 backend chain 만 보장. **Frontend UX intent 3 critical gap** 존재. v1.4 cascade 필요.

## 1. Critic 항목별 분석

### 1.1 "팝업창" 표현의 의미

v1.3 fix: `_blank` window.open → `location.assign` (same-tab nav).

**사용자 비판**: "팝업창으로 호출되는 방식 해결안됨".

**Critic 분석**:

| 가능성 | 신빙성 | 근거 |
|--------|:------:|------|
| (A) Browser cache: 옛 `_blank` JS 잔존 | ⭐⭐⭐⭐ | 검증 필요 — service worker 가 지속 |
| (B) `location.assign` 가 cross-origin 일 때 popup 처럼 보임 | ⭐⭐ | :3000→:3001 = same-host different-port, popup 처리 아님 |
| (C) 사용자가 의도한 UX = same-tab nav 자체가 wrong | ⭐⭐⭐⭐⭐ | "interaction in lobby" 명시 — lobby UI 보존 의도 |

**결론**: (C) 가 본질. 사용자는 **lobby UI 보존 + CC active 시각 표현**을 원함. v1.3 이 이를 spec 화 안 함.

### 1.2 "interaction in lobby" 의 정확한 의미

사용자 원문: "해당 로비에 해당 cc 를 선택해서 진행중이라는 상호작용이 추가되어야 함"

**Critic 해석 후보 3개**:

| 해석 | UX 형태 | 평가 |
|------|--------|------|
| α. 상태 pill | lobby 에 "CC for Table N — Active" indicator. CC 는 다른 tab/window | popup 으로 회귀 → ❌ |
| β. iframe embed | lobby 내 dialog/panel 에 CC iframe. lobby underneath 보존 | ⭐ 사용자 의도 가장 정합 |
| γ. SPA 내부 라우트 | lobby 내 /cc/{id} 화면 — 같은 Flutter 앱이 CC 화면 직접 렌더 | 큰 redesign (CC 코드 lobby 에 통합) — overkill |

**Default 채택**: β (iframe embed). **근거**:
- v1.3 의 cross-origin 문제 (lobby :3000 / cc :3001) 해소
- CC 는 독립 Flutter 앱 그대로 유지 (코드 분리 보존)
- iframe 내 CC 가 자체 라우터 + 자체 상태 보유
- Lobby UI 가 dialog 뒤에 보존 → 사용자 의도 ("active 표현") 정합
- 닫기 버튼으로 lobby 즉시 회귀

### 1.3 "connect 창은 비번 입력만" — 의미 분석

v1.3 connect form: tableId + token + wsUrl + boBaseUrl (4 fields).

**사용자 의도**: 비번 1 field. 나머지는 "lobby settings 에서 설정 가능".

**Critic 분석**:

| 가정 | 결과 |
|------|------|
| Lobby settings = 공용 설정 page (BO URL, default email, 등 저장) | Lobby 에 settings 화면 추가 필요 |
| CC 는 그 설정을 어떻게 받음? | URL query (lobby launch 시) + localStorage (영속) |
| CC stand-alone 진입 (browser 직접 :3001 접속) 시 | localStorage 에 last config + 비번만 입력 |
| 비번 입력 후 인증 흐름 | POST /auth/login → token → CC 사용 |

**결론**: stand-alone CC = (a) localStorage 에서 last config 읽기 (b) 비번 prompt (c) /auth/login 으로 token 갱신 (d) CC 사용. 4-field form 폐기.

### 1.4 v1.3 의 검증 수준 부족

v1.3 evidence: `tools/e2e_lobby_to_cc_ws.py` 9/9 PASS (backend chain).

**Critic**: 9/9 PASS = backend HTTP/WS endpoint 도달 PASS. **Frontend 실제 동작 미검증**:
- lobby 버튼 클릭 → repo.launchCc() 호출 → response 처리 → cc_launcher → location.assign 까지 실행 검증 0
- CC main() 의 Uri.base.queryParameters 파싱 동작 검증 0
- at_00_login_screen 의 placeholder vs form 분기 동작 검증 0

**SOP 위배**: "evidence before assertions". Backend evidence 만으로 frontend PASS 주장 = SOP §"verification before completion" 위반.

**보완 필요**: Playwright 등 brower-level E2E (단, 본 cycle 은 manual rebuild + screenshot 으로 대체).

## 2. v1.4 설계 (cascade)

### 2.1 Lobby iframe embed (issue 1)

**목표**: "Enter CC" 클릭 → lobby 내 fullscreen Dialog + iframe.

| 구성 | 위치 | 역할 |
|------|------|------|
| `cc_iframe_view.dart` (web/stub) | `team1-frontend/lib/features/lobby/widgets/` | HtmlElementView 로 iframe 임베드 |
| `_handleLaunchCc` | `lobby/screens/table_detail_screen.dart` | location.assign 폐기 → showDialog (Dialog.fullscreen) |
| Active CC 헤더 | Dialog AppBar | "Active CC — Table N" + 닫기 |
| iframe URL | response.cc_url | 기존 v1.3 endpoint 그대로 |

**Cross-origin 정책**: lobby :3000 → iframe :3001. Same-host different-port 는 iframe 가 정상 작동. CSP 미설정 시 default allow.

**Iframe sandbox**: 미적용 (CC 가 lobby 와 통신 안 함, fully isolated).

### 2.2 CC password-only connect form (issue 2)

**목표**: stand-alone 진입 시 비번 1 field.

| Source | Field | 동작 |
|--------|-------|------|
| URL query / args | tableId, token, ccInstanceId, wsUrl, boBaseUrl | 자동 로드 (v1.3 그대로) |
| localStorage | 위 + email (last successful login) | URL 없을 때 fallback |
| 사용자 입력 | password | /auth/login → 새 token |

**Form 변경**:
- 4-field → 1-field (Password)
- 상단: "Last session: BO=..., Table=..." (read-only, localStorage 에서)
- localStorage 비어 있으면 "[Configure]" 버튼 → 4-field expanded form (advanced)

**localStorage 키**: `ebs_cc_last_config` = `{boBaseUrl, email, tableId, wsUrl}` (token 제외 — 짧은 수명)

### 2.3 Cache bust 전략

Flutter web 이 새 빌드 시 `main.dart.js` 해시 자동 갱신 → service worker 가 새 asset 다운로드. 단:
- `index.html` 의 `<base>` + `flutter_bootstrap.js` 가 캐시될 수 있음
- `flutter_service_worker.js` 자체가 캐시되어 새 SW 등록 안 될 가능성

**보장 방법**:
1. 빌드 시 `--release` (default)
2. Container restart (이미 자동)
3. Browser 측 force-refresh (Ctrl+Shift+R) 사용자 안내
