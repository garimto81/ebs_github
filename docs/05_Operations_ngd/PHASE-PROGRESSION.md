## 2. 각 Phase 상세

### 2.1 Phase 0 (업체 선정 + 분석)

**목표:** RFID 업체를 선정하고 PokerGFX 소프트웨어 분석을 완료합니다.

**주요 활동:**

| 활동 | 상세 | 담당 | 상태 |
|------|------|:----:|:----:|
| 업체 리서치 | PCB/SMT/안테나 업체 조사 | - | ✅ |
| 업체 관리 시스템 | 업체 추적 문서 작성 | - | ✅ |
| 견적 요청 | 1순위 업체 RFQ 발송 | - | 🟡 |
| 견적 비교 | 비용/납기/품질 비교 | - | ⏳ |
| 업체 선정 | 최종 업체 결정 | - | ⏳ |
| 계약/발주 | 계약서 체결, 발주서 발행 | - | ⏳ |
| PokerGFX 분석 | 소프트웨어 역공학 분석 | - | ✅ |

**완료 조건:**

- [ ] 1개 이상 업체와 계약 완료
- [ ] 프로토타입 발주 완료
- [x] PokerGFX PRD v3.0 완료 (95% 커버리지, 8모듈/2,887 .cs 파일)

**주요 리스크:**

| 리스크 | 확률 | 영향 | 대응 |
|--------|:----:|:----:|------|
| 견적 지연 | 중 | 중 | 복수 업체 동시 문의 |
| ST25R3911B 재고 없음 | 낮 | 높 | 대안 부품 조사 |
| 예상 비용 초과 | 중 | 중 | 스펙 조정, 수량 조정 |

### 2.2 Phase 1 (GFX 복제)

**목표:** PokerGFX의 모든 기능을 100% 복제합니다.

**구성:**

```
Phase 1 내부 구조:

Phase 1-1: RFID 기초
├── MFRC522 테스트 환경 구축
├── 카드 UID 인식 테스트
└── MCU-Server 통신 구현

Phase 1-2: 서버 구현
├── Server 개발 (미정)
├── WebSocket 구현
├── SQLite DB 구축
└── 카드 매핑 API

Phase 1-3: ST25R3911B 검증
├── 프로덕션 RFID 모듈 테스트
├── 안테나 성능 검증
└── 10-Reader 구성 테스트

Phase 1-4: Frontend
├── Flutter UI 개발
├── 방송 오버레이 (Rive) 구현
├── Action Tracker 완성
└── Viewer Overlay 완성

Phase 1-5: 통합 테스트
├── 전체 기능 테스트
├── 4시간 연속 운영 테스트
├── 버그 수정 및 안정화
└── 운영자 교육
```

**완료 조건:**

| # | 조건 | 측정 방법 | 목표값 |
|:-:|------|----------|:------:|
| 1 | 기능 복제율 | 체크리스트 완료율 | 100% |
| 2 | RFID 인식률 | 5장 카드 테스트 | 100% |
| 3 | E2E 지연 | 카드→화면 시간 | < 200ms |
| 4 | 연속 운영 | 무중단 테스트 | 4시간+ |
| 5 | ST25R3911B 검증 | SPEC 문서 항목 | 10/10 PASS |
| 6 | 운영자 승인 | 실사용 테스트 | 승인 |

**핵심 마일스톤:**

| # | 마일스톤 | 검증 방법 |
|:-:|----------|----------|
| 1 | MFRC522 카드 인식 성공 | Serial 출력 UID 확인 |
| 2 | 서버 MVP 완성 | WebSocket 카드 이벤트 |
| 3 | ST25R3911B 동작 확인 | 프로덕션 모듈 테스트 |
| 4 | UI 베타 완성 | 내부 시연 |
| 5 | 4시간 무중단 통과 | 스트레스 테스트 |

Phase 2 진입 가능 조건:

✅ Phase 0 완료

   └── 프로토타입 수령

✅ Phase 1 완료
   ├── 기능 100% 복제
   ├── 4시간 무중단 통과

   └── "Phase 2 시작 가능" 결정

| 블라인드 연동 | 블라인드 레벨 자동 업데이트 | 5% |
| 히스토리 동기화 | 핸드 히스토리 자동 저장 | 5% |
| 충돌 해결 UI | 데이터 불일치 시 수동 선택 | 5% |

**완료 조건:**

| # | 조건 | 측정 방법 | 목표값 |
|:-:|------|----------|:------:|
| 1 | DB 연동 | 데이터 동기화 성공 | 정상 |
| 2 | 수동 입력 감소 | 운영 로그 분석 | 30%+ |

Phase 3 진입 가능 조건:

| 2 | 정확도 | 자동 계산 오류율 | < 1% |
| 3 | 안정성 | 연속 운영 테스트 | 12시간+ |

## 4. 의사결정 프로세스

### 4.1 Phase 전환 승인

```
Phase 전환 승인 프로세스:

1. 완료 조건 체크
   └── 체크리스트 모든 항목 완료

2. 검증 테스트
   └── 정의된 테스트 모두 통과

3. 문서 작성
   └── 완료 보고서 작성

4. 팀 리뷰
   └── 관계자 리뷰 및 피드백

5. 최종 승인
   └── 책임자 승인

6. Phase 전환
   └── 다음 Phase 시작
```

### 4.2 예외 처리

**조건 미충족 시:**

| 상황 | 대응 |
|------|------|
| 조건 1-2개 미충족 | 해당 조건만 완료 후 전환 |
| 50% 이상 미충족 | Phase 연장, 원인 분석 |
| 외부 요인 (납기 등) | 병렬 진행 또는 대안 검토 |

**긴급 상황:**

```
Phase 1 미완료 시 대응 전략:

Option A: 핵심 기능만 우선 완료
├── P0 기능만 구현
├── P1/P2 기능 후순위
└── 수동 운영 병행

Option B: 부분 전환
├── 가능한 기능만 EBS 사용
├── 나머지는 수동 운영
└── 점진적 마이그레이션
```
